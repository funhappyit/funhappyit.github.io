## 문제 상황

Vue 3 + DevExtreme 조합으로 만든 관리자 화면에서, 목록 그리드 하단에 데이터가 없는 빈 공간이 남는 현상을 겪었다.

- 페이지네이션이 걸려 있는 그리드인데, 검색 결과 총 건수가 충분히 많은데도 그리드 하단에 빈 공간이 보였다.
- "데이터가 많으면 꽉 차야 하는 거 아닌가?"라는 직관과 반대되는 현상이라 처음엔 원인이 잘 안 보였다.
- 반면 같은 그리드 컴포넌트를 쓰는 다른 화면은 별다른 설정 없이도 항상 꽉 차 보였다.

DevExtreme(또는 비슷한 방식으로 동작하는 다른 그리드 라이브러리)을 쓸 때 흔히 겪을 수 있는 문제라, 원인과 정리 과정을 남겨둔다.

## 원인

### 1. `height`는 "고정값"이지 "콘텐츠 기반 자동 축소"가 아니다

DevExtreme DataGrid에 `height`를 명시적으로 지정하면, 그 값은 실제 데이터 행 개수와 무관하게 항상 그대로 유지된다. 행이 몇 개든 컨테이너 크기는 지정한 값 그대로다.

```js
// 그리드를 감싸는 래퍼 컴포넌트에서 흔히 쓰는 패턴
function resolveHeight(explicitHeight) {
  if (explicitHeight == null) {
    // height를 안 주면 뷰포트 기준으로 자동 계산
    return `calc(100vh - ${getTopOffset()}px)`;
  }
  // height를 주면 그 값을 그대로 사용 (콘텐츠 크기와 무관)
  return explicitHeight;
}
```

여기서 중요한 건 두 경우 모두 **"행 개수 기반 축소"는 지원하지 않는다**는 점이다. `height`를 안 주면 뷰포트 기준으로 계산될 뿐이지, 콘텐츠(행 수)에 맞춰 줄어드는 게 아니다.

### 2. 페이지 사이즈는 총 건수와 무관하게 화면에 보이는 행 수를 잘라버린다

```js
const gridOptions = {
  height: '80vh',       // 뷰포트의 80%로 고정
  paging: {
    enabled: true,
    pageSize: 20,        // 총 건수와 무관하게 한 페이지엔 최대 20건만
  },
};
```

총 검색 결과가 아무리 많아도, 실제로 한 번에 렌더링되는 행은 **pageSize만큼**뿐이다. 그런데 고정 `height`가 그 pageSize만큼의 행이 차지하는 실제 픽셀 높이보다 넉넉하게 잡혀 있으면, **페이지가 꽉 차 있어도** 하단에 빈 공간이 남는다. 데이터가 적은 페이지(검색 결과가 적거나, 필터링 후 마지막 페이지 등)라면 당연히 더 크게 비어 보인다.

정리하면 빈 공간의 크기를 좌우하는 건 "총 데이터 건수"가 아니라 **"pageSize"와 "고정 height 값의 크기 차이"** 였다.

### 3. 반대로 문제없던 화면은 왜 괜찮았나

문제없이 항상 꽉 차 보이던 화면들의 공통점은 하나였다 — **`height`도, 페이징 관련 설정도 아예 건드리지 않았다.** 자동 계산에 그대로 맡기고, 데이터도 페이징 없이 필요한 만큼 렌더링되는 구조라 "정해진 슬롯 수보다 데이터가 적어서 빈칸이 남는" 상황 자체가 발생할 수 없었다.

## 해결 방법 (그리고 무작정 지우면 안 되는 경우들)

기본 방향은 단순하다 — **불필요하게 고정된 `height`를 지우고 자동 계산에 맡긴다.** 다만 실제로 여러 화면에 이 원칙을 적용해보면, 아래 몇 가지 경우는 예외로 남겨둬야 한다는 걸 알게 됐다.

### 예외 1 — 모달/팝업 내부 그리드

모달은 크기가 제한된 다이얼로그이기 때문에, 화면 전체를 채우는 자동 계산으로 바뀌면 오히려 레이아웃이 깨진다. 모달 안의 그리드는 고정 height를 그대로 두는 게 맞다.

### 예외 2 — 고정 비율 분할 레이아웃 안의 그리드

```html
<!-- 화면을 위/아래로 고정 비율 분할하는 레이아웃 -->
<div class="h-2/5">
  <data-grid height="100%" />
</div>
<div class="h-3/5">
  <data-grid height="100%" />
</div>
```

이런 화면에서 그리드의 `height`를 지우면, 그리드가 뷰포트 전체 높이만큼 늘어나려고 해서 위/아래 그리드가 겹치거나 레이아웃이 깨질 수 있다. 부모 컨테이너가 `flex: 1 1 auto`처럼 유동적인 경우와, `h-2/5`처럼 고정 비율인 경우를 구분해야 한다.

### 예외 3 — 이미 다른 이유로 계산된 고정값인 경우

`height`가 단순히 방치된 게 아니라, 옆에 나란히 배치된 다른 위젯(달력, 사이드 패널 등)의 실제 높이에 맞추기 위해 의도적으로 계산해서 넣은 값일 수도 있다. 이런 경우 "auto로 바꾸면 콘텐츠 크기로 안 줄어든다"는 걸 이미 확인한 뒤 대안으로 고정값을 쓴 것이므로, 겉보기엔 똑같이 "하드코딩된 height"라도 원인과 목적이 완전히 다르다. 무작정 일괄 제거하면 이번엔 옆 위젯과 높이가 어긋나는 새로운 버그를 만들게 된다. 그래서 값을 지우기 전에 **왜 이 값이 이 숫자로 박혀 있는지**부터 확인하는 절차가 꼭 필요했다.

### 함정 — JS와 CSS에 높이가 중복 지정된 경우

```js
// JS: 그리드 자체의 height
const gridOptions = { height: 'calc(100vh - 200px)' };
```

```css
/* CSS: 그리드를 감싸는 컨테이너에도 별도로 고정 높이가 걸려 있음 */
.grid-wrapper {
  height: calc(100vh - 220px);
  overflow: hidden;
}
```

이 경우 JS 쪽 `height`만 지워서는 문제가 해결되지 않는다. 바깥 컨테이너가 `overflow: hidden`과 함께 별도의 고정 높이를 한 번 더 강제하고 있으면, 그리드 자체는 콘텐츠 크기로 계산되려 해도 부모가 그 공간을 그대로 유지한 채 잘라버리기 때문이다. JS만 고치고 "해결됐다"고 판단했다가 실제 화면은 그대로였던 경우다. 이런 케이스는 JS 옵션뿐 아니라 감싸는 CSS까지 함께 확인해야 한다.

## 결과 및 체크리스트

- 그리드 자체의 `height` 지정만 보고 판단하지 말고, ① 페이징 사용 여부, ② 부모 레이아웃이 고정 비율인지, ③ 이미 다른 목적으로 계산된 값은 아닌지, ④ 감싸는 컨테이너에 중복된 CSS height가 있는지를 먼저 확인한다.
- 위 네 가지에 해당하지 않는, 단순히 방치되어 있던 고정 height라면 지우고 자동 계산에 맡기는 게 안전하다.
- 흥미로웠던 건 "단순한 화면일수록 애초에 height를 안 건드려서 문제가 없었고, 복잡한 레이아웃일수록 개발 당시 수동으로 값을 맞추다가 문제가 생겼다"는 패턴이었다. 그래서 이런 버그를 찾을 때는 오히려 단순한 화면보다 여러 위젯이 얽힌 복잡한 화면들을 먼저 의심하는 게 효율적이었다.
